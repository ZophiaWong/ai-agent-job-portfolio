# Agent 辅助开发操作检查表

这份表用于一次受控任务。它不替代需求判断，也不把 Agent 的说明当作验证结果。

## 开始前

- [ ] 写下目标、验收条件、允许改动的范围和禁止修改项。
- [ ] 标出数据、权限、外部服务、密钥、删除或不可逆操作。
- [ ] 为高风险写操作指定人工确认人、回滚方式和执行环境。
- [ ] 记录要读取的 README、入口、测试、日志和当前 diff；每项写明来源。

## 只读探索与计划

- [ ] 让 Agent 先只读查找相关文件、现有模式和可运行的检查。
- [ ] 要求它区分源码事实、测试覆盖、文档说明和自己的推测。
- [ ] 计划逐个列出要改的文件、理由、保留行为、测试和风险。
- [ ] 计划出现无关重构、新依赖或范围不明时，先缩小任务。

## 最小 patch

- [ ] 只改已批准的文件；不要混入格式化、重命名或另一个需求。
- [ ] 修改前后都查看 diff，避免覆盖用户未提交的内容。
- [ ] 涉及权限时补负向 case；涉及写操作时检查幂等、超时和回滚。
- [ ] Agent 提到文件、接口或依赖时，回到仓库核对，不能凭解释接受。

## 验证与 review

- [ ] 运行与验收条件匹配的测试、类型检查、lint 或手动复现步骤。
- [ ] 检查失败输出；不要把“命令运行过”误写成“验证通过”。
- [ ] 逐个阅读 diff：范围是否一致、公共行为是否变化、有没有调试内容或敏感信息。
- [ ] 人工确认高风险动作和残余风险，再决定提交、回退或继续修改。

## bad case 回流

```text
任务与验收：
失败现象：
证据：失败测试、日志或 diff
漏检层级：需求 / 上下文 / 工具 / 代码 / 测试 / review
下次要加的约束或检查：
```

记录具体事实。比如“Agent 引用了不存在的接口，测试在导入阶段失败”，比“模型幻觉了”更能帮助下次排查。
