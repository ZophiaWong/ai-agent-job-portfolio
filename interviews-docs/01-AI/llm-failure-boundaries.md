# LLM 失败边界

主要能力：`C02.01`。按[统一练习协议](../practice-protocol.md)完成一次冷答、迁移和延迟复测。

## 冷启动：先分类，再选处理方式

不看后文，用 8 分钟处理下面三条故障。对每条回答：

1. 这是什么故障模式；
2. 主要补救放在模型前、模型调用时，还是服务端后处理；
3. 什么情况下应拒答、弃答或升级给人。

| 案例 | 现象 |
| --- | --- |
| A | 同一输入调用多次，分类标签偶尔从 `approved` 变成 `approve`，即使 `temperature=0` 也出现过一次。 |
| B | 模型返回的 JSON 缺少 `risk_level`，但调用方直接把它当成成功结果。 |
| C | 用户让 Agent 操作一个无法确认所有者的账户，模型语气很确定。 |

把结果写入 `llm-boundary-cold.md`。表格中至少要有“分类”、“可检查信号”、“选定的处理”和“为什么不选其他方法”四列。这是确定性检查任务：只要案例和表格完整，就能逐项复查，不需要运行模型。

## 迁移题

把 A 改成“输出格式始终正确，但风险等级在 `low` 和 `high` 之间波动”。重新选择处理方式，说清哪些检查还能用，哪些已经不够。

## 延迟复测与证据边界

隔天只看三个案例，5 分钟重做分类和处理选择，存为 `llm-boundary-retest.md`。阅读完参考区不算掌握证据；在提示后改对的答案也不算独立证据。

## 参考边界

### 随机输出与非确定性

LLM 的输出存在随机性。降低 `temperature` 可以收窄采样范围，但不应把 `temperature=0` 解读为端到端确定性保证：服务端实现、并发批处理和模型版本变化都可能让结果仍然不确定。对稳定输出的要求应落在服务端契约和测试上。

### 结构化输出不是信任了输出

对输出做 `schema` 验证，检查必填字段、枚举值、长度和跨字段约束。验证失败时返回明确错误，不要默认补成“成功”。大小写归一、白名单映射和数值截断属于确定性后处理；它们只能修正已定义的差异，不能替模型猜缺失的业务事实。

### 拒答、不确定和升级

- 缺少必要输入时，先澄清或拒答。
- 可回答但证据弱时，显示不确定和证据范围。
- 涉及高风险副作用、所有权不清或政策冲突时，弃答并升级给有权限的人。

### 评分要点

| 等级 | 可观察表现 |
| --- | --- |
| 0–1 | 只列“重试、换模型、改 prompt”，没有识别故障。 |
| 2 | 能区分格式错误和业务风险，但处理方式过于笼统。 |
| 3 | 根据故障信号选择 schema 验证、确定性后处理、弃答或升级，并能排除不合适的方法。 |
| 4 | 约束变化后仍能调整方案，并提出可重现的测试、监控和人工门槛。 |

这些内容是通用练习，不代表任何个人项目已实现了上述控制。
