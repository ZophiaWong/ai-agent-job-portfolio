## 1. 本章学习目标

本章把前面的知识转成项目表达。面试中，真正拉开差距的不是“知道 RAG 是什么”，而是能讲清：

```text
为什么做这个项目；
用户场景是什么；
系统架构怎么设计；
RAG、工具、Workflow、状态、评测、安全如何组合；
上线后怎么监控；
失败时怎么降级；
项目指标如何衡量。
```

每个项目模板都按如下结构组织：

```text
业务目标

安全策略

面试表达版本
```

## 2. 项目模板一：企业知识库 RAG Agent

### 2.1 业务目标

帮助企业员工基于内部文档、制度、产品手册、技术文档快速获得可引用答案，降低人工咨询成本。

### 2.2 系统架构

```text
用户问题
  ↓
FastAPI
  ↓
身份认证 / 权限识别
  ↓
Query Rewrite
  ↓
知识库检索
  ├── 向量检索
  ├── BM25 检索
  └── 元数据权限过滤
  ↓
Hybrid 合并
  ↓
Rerank
  ↓
上下文压缩
  ↓
LLM 生成答案
  ↓
引用校验
  ↓
低置信度处理
  ├── 补充检索
  ├── 返回不确定说明
  └── 转人工
  ↓
答案 + 引用来源
```

### 2.3 关键模块

```text
文档索引模块：
  解析、清洗、切块、embedding、索引版本。

在线问答模块：
  query rewrite、hybrid retrieval、rerank、answer generation。

权限模块：
  用户身份、部门、文档 ACL、检索前过滤。

评测模块：
  Recall@K、引用准确率、答案正确率、用户反馈。

监控模块：
  空召回率、p95 延迟、token 成本、bad case。
```

### 2.4 面试表达版本

```text
我做的企业知识库 Agent 不是简单向量库问答，而是分成离线索引和在线检索两条链路。离线侧负责文档解析、切块、embedding、元数据和索引版本；在线侧先做 query rewrite，再进行向量和 BM25 混合召回，经过 rerank 和上下文压缩后交给模型生成，并强制带引用。为了生产可用，我加入了权限过滤、引用校验、空召回降级、bad case 回流和检索评测。
```

## 3. 项目模板二：客服 Agent

### 3.1 业务目标

自动处理高频客服问题，辅助人工客服，降低响应时间，并在低置信度或高风险场景转人工。

### 3.2 系统架构

```text
用户咨询
  ↓
意图识别
  ↓
风险判断
  ├── 投诉 / 退款 / 账户安全 → 转人工或审批
  └── 普通问题 → 自动处理
  ↓
知识库 RAG
  ↓
订单 / 用户信息只读工具
  ↓
答案生成
  ↓
置信度判断
  ├── 高置信 → 回复
  └── 低置信 → 推荐话术 + 转人工
  ↓
会话摘要和工单记录
```

### 3.3 关键设计

```text
工具：
  订单查询、物流查询、政策查询、工单创建。

安全：
  用户身份校验，敏感信息脱敏，高风险动作不自动执行。

状态：
  会话阶段、用户诉求、已查询订单、人工接管状态。

评测：
  首响时间、自动解决率、转人工率、用户满意度、错误回复率。
```

### 3.4 面试表达版本

```text
客服 Agent 的关键不是让模型自由回答，而是用意图识别和风险分级控制路径。普通 FAQ 走 RAG，高风险问题转人工，涉及订单信息的工具只读并且做用户身份校验。系统会记录会话状态和工具调用 trace，低置信度时不强答，而是生成推荐话术或转人工。
```

## 4. 项目模板三：代码助手 Agent

### 4.1 业务目标

帮助开发者理解代码库、定位问题、生成修改建议、解释调用链和辅助代码审查。

### 4.2 系统架构

```text
用户问题 / 代码任务
  ↓
任务分类
  ├── 理解代码
  ├── 定位 bug
  ├── 生成修改建议
  └── 审查 diff
  ↓
代码库索引
  ├── 文件摘要
  ├── 函数 / 类切块
  ├── AST / 调用图
  └── Git diff
  ↓
检索相关代码
  ↓
上下文组装
  ↓
LLM 分析
  ↓
可选工具
  ├── 运行测试
  ├── 静态检查
  └── 生成 patch
  ↓
输出解释 / 修改建议 / 测试建议
```

### 4.3 关键设计

```text
RAG：
  代码不能只按固定 token 切块，要按函数、类、文件和依赖关系切。

工具：
  grep、AST 解析、测试运行、lint、git diff。

安全：
  默认只读，生成 patch 不自动提交，执行代码必须沙箱。

评测：
  定位准确率、patch 可用率、测试通过率、误修改率。
```

### 4.4 面试表达版本

```text
代码助手 Agent 的核心是代码上下文工程。我会建立文件级、函数级和模块级索引，并结合关键词检索、向量检索和调用图。Agent 先检索相关代码，再生成分析或 patch。涉及运行测试或执行脚本时放到沙箱里，默认不自动提交修改。评测上关注 patch 是否通过测试、定位是否准确和是否引入副作用。
```

## 5. 项目模板四：多 Agent 研究助手

### 5.1 业务目标

针对复杂主题自动拆解问题、收集资料、交叉验证、生成研究报告。

### 5.2 架构

```text
用户研究目标
  ↓
Planner Agent：拆解子问题
  ↓
Research Agents：并行收集资料
  ↓
Evidence Aggregator：合并证据
  ↓
Critic Agent：检查证据支持和矛盾
  ↓
Writer Agent：生成报告
  ↓
Citation Validator：校验引用
  ↓
最终报告
```

### 5.3 风险和治理

```text
风险：
  搜索结果质量不稳定、引用错误、过度总结、成本高。

治理：
  最大搜索轮数、来源可信度排序、引用校验、报告结构模板、人工复核。
```

### 5.4 面试表达版本

```text
多 Agent 研究助手适合体现任务分解和角色协作。我会用 Supervisor 调度 Planner、Researcher、Critic 和 Writer。每个 Agent 有明确输入输出，证据统一写入 State，最终报告必须带来源。为了防止循环和成本失控，我会限制搜索轮数、并行度和 token budget。
```

## 6. 项目模板五：数据分析 Agent

### 6.1 业务目标

用户用自然语言提出数据问题，Agent 查询数据、生成分析、绘制图表并解释结论。

### 6.2 架构

```text
用户问题
  ↓
意图识别
  ↓
权限校验
  ↓
SQL 生成 / 工具选择
  ↓
SQL 安全检查
  ↓
只读查询
  ↓
结果摘要
  ↓
可选 Python 分析
  ↓
图表生成
  ↓
结论解释
```

### 6.3 关键设计

```text
SQL 安全：
  只允许 SELECT，只访问授权表，限制行数。

工具：
  数据字典检索、SQL 查询、Python 分析、图表生成。

评测：
  SQL 正确率、查询安全率、结论准确率、图表可读性。

降级：
  SQL 不确定时生成查询建议，不自动执行。
```

## 7. 项目模板六：高风险业务审批型 Agent

```text
用户请求
  ↓
Agent 生成执行计划
  ↓
风险等级判断
  ↓
低风险：自动执行
  ↓
中风险：用户确认
  ↓
高风险：人工审批
  ↓
审批通过后冻结参数
  ↓
幂等执行
  ↓
审计日志
```

适合场景：

```text
邮件发送；
数据库写操作；
生产配置变更；
工单提交；
合同生成；
对外发布。
```

面试表达：

```text
对于有副作用的 Agent，我不会追求全自动，而是按风险分级。模型生成计划和参数，后端做权限和合法性校验，高风险动作进入 HITL。审批前展示执行内容，审批后冻结参数并通过幂等键执行，所有动作写审计日志。
```

## 8. 通用项目表达模板

```text
1. 背景：
   业务中存在什么痛点？

2. 目标：
   要提升什么指标？

3. 架构：
   Agent、RAG、工具、状态、后端、存储如何组合？

4. 核心难点：
   检索、上下文、工具、评测、安全哪个最难？

5. 解决方案：
   你具体做了什么设计？

6. 指标：
   如何证明有效？

7. 风险：
   可能失败在哪里？

8. 迭代：
   如何通过 Bad Case 回流持续改进？
```

## 8. 面试题与答题闭环
### Q：如何设计一个企业知识库 RAG Agent？

**考察点：** RAG 项目系统设计。

**推荐回答：** 我会分离线索引和在线问答两条链路。离线做解析、清洗、切块、embedding、元数据和版本管理；在线做权限过滤、query rewrite、hybrid search、rerank、上下文压缩、答案生成和引用校验。生产上还要监控空召回、引用准确率、延迟、成本和用户反馈。

### Q：如何把 Agent 项目讲得有工程深度？

**考察点：** 项目表达能力。

**推荐回答：** 不要只说用了某个模型或框架，而要讲业务目标、架构分层、核心模块、状态流、工具流、评测指标、安全策略和降级方案。尤其要说明失败时怎么排查、上线后怎么监控。

### Q：代码助手 Agent 的关键难点是什么？

**考察点：** 代码 Agent 设计。

**推荐回答：** 关键在代码上下文工程和安全执行。需要按函数、类、文件、调用关系建立索引，结合关键词、向量和调用图检索；生成修改建议后不能自动提交，运行代码或测试必须在沙箱中执行。
