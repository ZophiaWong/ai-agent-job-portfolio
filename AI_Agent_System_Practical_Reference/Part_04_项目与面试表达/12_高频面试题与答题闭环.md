## 1. 使用说明

本章不是简单背答案，而是提供“答题闭环”。每道题都建议按下面结构回答：

```text
先给定义；
再讲为什么需要；
再讲核心机制；
再讲工程实现；
再讲常见坑和权衡；
最后给项目或生产化例子。
```

## 2. Agent 基础类

### Q1：什么是 AI Agent？

**推荐回答：**

AI Agent 是以 LLM 为决策核心，结合任务规划、工具调用、状态管理、记忆和反馈机制，在一定边界内自动或半自动完成任务的软件系统。它和普通 Chatbot 的区别是：Chatbot 主要生成文本，Agent 需要在多步骤流程中选择工具、执行动作、读取 Observation、更新状态并决定下一步。工程上，Agent 不是一个模型调用，而是 Orchestrator、LLM、Workflow、Tool Layer、State、Memory、安全和评测组成的系统。

**追问方向：**

```text
Agent 和 Workflow 的区别？
Agent 为什么难以上线？
Agent 如何避免循环？
```

### Q2：Agent 和普通 LLM Chatbot 有什么区别？

**推荐回答：**

Chatbot 通常是用户输入到模型输出的短链路，重点是回答质量。Agent 以任务完成为目标，可能需要规划、检索、调用工具、操作外部系统、等待人工审批和恢复执行。Agent 的核心风险也更大：不仅可能回答错误，还可能误调用工具、越权访问或重复执行副作用操作。因此 Agent 工程重点是状态机、工具治理、权限、安全、评测和监控。

### Q3：ReAct 是什么？

**推荐回答：**

ReAct 是 Reasoning + Acting 的思想，即模型在推理和行动之间交替：先判断下一步，再调用工具，观察结果后继续修正。生产实现中，我会记录结构化 Action 和 Observation，而不是把内部思考原文暴露给用户。为了避免循环，需要设置最大步数、终止条件、工具错误处理和状态记录。

## 3. Workflow 与多 Agent 类

### Q4：Chain、Workflow、Agent 有什么区别？

**推荐回答：**

Chain 是固定线性步骤，适合简单可预测任务；Workflow 是显式状态机，支持分支、循环、错误处理和人工审批；Agent 是模型动态决定下一步。生产系统通常不是完全自治，而是 Workflow + 局部 Agent 决策的混合形态，用代码保证边界，用模型处理开放判断。

### Q5：为什么生产系统常用 LangGraph 这类框架？

**推荐回答：**

因为 Agent 任务通常是 long-running、stateful、多步骤、有分支、有工具调用、有人工审批和失败恢复需求。LangGraph 通过 State、Node、Edge、Conditional Edge 和 Checkpoint，把 Agent 执行过程显式化，便于控制、调试、恢复和观测。相比一串自由模型调用，状态机框架更适合生产治理。

### Q6：多 Agent 有什么优势和问题？

**推荐回答：**

优势是角色分工、并行处理、交叉审查和复杂任务拆解。例如 Planner 负责计划，Researcher 负责检索，Tool Agent 负责执行，Critic 负责审查。问题是通信成本高、状态一致性难、容易循环、上下文污染和成本增加。工程上要有 Supervisor、结构化消息、共享状态、最大轮次和 trace。

## 4. RAG 类

### Q7：RAG 的完整流程是什么？

**推荐回答：**

RAG 分离线和在线两条链路。离线链路包括文档接入、解析、清洗、切块、Embedding、写入向量库和元数据索引、索引版本管理。在线链路包括用户问题、权限识别、Query Rewrite、向量召回/BM25 召回、Hybrid 合并、Rerank、上下文压缩、LLM 生成、引用校验和日志回流。

### Q8：RAG 召回不准怎么优化？

**推荐回答：**

先判断失败在哪一层：如果正确证据没召回，查 query rewrite、chunking、embedding、元数据过滤和索引新鲜度；如果召回了但排序差，查 hybrid search、rerank 和 TopK；如果证据在上下文里但模型没用，查上下文压缩、prompt、引用约束和答案校验。不要一上来只调 TopK 或换模型。

### Q9：Chunking 怎么设计？

**推荐回答：**

Chunking 是召回粒度和语义完整性的权衡。块太大噪声高、成本高，块太小语义断裂。可以按标题、段落、语义边界切块，并设置适度 overlap。代码按函数、类和模块切，表格要保留表头和行列语义。复杂场景可用 small-to-large 或多尺度切块。

### Q10：为什么需要 Rerank？

**推荐回答：**

初召回阶段追求召回率，会得到较多候选；Rerank 阶段用更精细的模型或规则对候选排序，提高最终进入上下文的证据质量。向量检索可能语义相关但不精确，BM25 可能词面匹配但语义弱，Rerank 可以综合判断 query 和文档的真实相关性。

## 5. 上下文与记忆类

### Q11：Prompt Engineering 和 Context Engineering 有什么区别？

**推荐回答：**

Prompt Engineering 关注指令怎么写，包括角色、格式、约束和示例；Context Engineering 关注哪些信息进入模型，包括历史摘要、RAG 证据、工具结果、状态、长期记忆如何选择、排序、压缩和淘汰。Agent 系统中，Context Engineering 往往比单纯 Prompt 更关键。

### Q12：Agent 的 State 和 Memory 有什么区别？

**推荐回答：**

State 是当前任务运行状态，例如计划、步骤、工具结果、错误、审批状态；Memory 是跨任务或跨会话可复用的信息，例如用户偏好、项目背景和历史经验。State 关注当前任务可恢复，Memory 关注长期复用。二者都不能无节制注入上下文。

### Q13：长期记忆如何避免污染？

**推荐回答：**

不能把所有对话都写入长期记忆。要先进入候选区，按稳定性、置信度、可复用性和权限判断是否写入。写入前去重和冲突检测，写入后支持 TTL、版本和删除。读取时只取与当前任务强相关且高置信的记忆。

## 6. Prompt / RAG / 微调类

### Q14：什么时候用 Prompt，什么时候用 RAG，什么时候微调？

**推荐回答：**

格式、语气、边界和简单约束问题优先 Prompt；事实知识、私有数据、时效性和可追溯性问题优先 RAG；当 Prompt、RAG、工具链路和 Workflow 都优化后，模型在长期行为、格式或偏好上仍不稳定，再考虑 SFT、LoRA 或 DPO。

### Q15：LoRA 和 QLoRA 的区别是什么？

**推荐回答：**

LoRA 冻结基座模型，只训练低秩增量矩阵，降低微调成本；QLoRA 在量化基座模型基础上训练 LoRA adapter，进一步降低显存需求。它们适合行为、格式和领域表达适配，但不适合替代 RAG 存储频繁变化的知识。

## 7. 工具调用与 MCP 类

### Q16：Function Calling 的完整流程是什么？

**推荐回答：**

模型根据工具描述输出工具名和参数，后端解析后进行 schema 校验、权限校验、幂等检查和风险判断，再执行真实工具。工具结果标准化为 Observation 回填给模型，模型决定继续调用、修正或输出最终答案。模型不直接执行工具，执行权在后端。

### Q17：如何保证工具调用安全可靠？

**推荐回答：**

工具可靠性包括工具描述、schema 校验、参数合法性、权限检查、超时、重试、熔断、fallback、错误码、幂等和审计。只读工具和写工具要分开治理，写工具必须有幂等键，高风险工具要人工审批。

### Q18：MCP 的作用是什么？

**推荐回答：**

MCP 是模型和外部工具/资源之间的标准化协议层，用于规范工具发现、schema、调用和资源暴露。Function Calling 更偏模型如何表达调用意图，MCP 更偏工具如何标准化暴露和治理。最终执行仍应由后端按权限和审计约束完成。

## 8. Python 后端与生产化类

### Q19：Python 后端如何实现 Agent 服务？

**推荐回答：**

我会用 FastAPI 做服务入口，Pydantic 做参数校验，LangGraph 做 Agent Workflow 编排，工具层封装 RAG、API、SQL 和代码执行，Redis 做缓存、锁、短期状态，PostgreSQL 保存任务、审计和权限，向量库做知识检索，并加入日志、指标、Trace、限流、熔断、降级和幂等。

### Q20：asyncio 和任务队列分别适合什么？

**推荐回答：**

asyncio 适合单个请求内的 I/O 并发，比如并行检索多个数据源或同时调用多个只读 API；任务队列适合跨请求的长任务，如文档索引、批量评测、复杂研究任务和可重试后台处理。长任务通常返回 task_id，后台 worker 执行。

### Q21：Agent 成功率下降怎么排查？

**推荐回答：**

我会按 Workflow、Tool、RAG、Context/Prompt、Model 分层排查。先看是否循环、超时、路由错误；再看工具是否 5xx、429、schema、权限错误；再看 RAG 是否空召回、错召回、索引过期、rerank 异常；然后看上下文是否过长、prompt 是否变更；最后看模型版本、参数和供应商行为变化。

## 9. 安全风控类

### Q22：Agent 安全和普通 Chatbot 安全有什么区别？

**推荐回答：**

普通 Chatbot 主要风险是有害输出和幻觉；Agent 还会调用工具、访问数据、执行代码和触发业务动作，所以风险包括越权、数据泄露、工具滥用、重复副作用和生产变更。治理必须包括权限、工具白名单、参数校验、沙箱、HITL、审计和监控。

### Q23：如何治理高风险工具？

**推荐回答：**

我会按风险分级。只读工具可自动执行，低风险写操作要求幂等，高风险动作进入 HITL，审批前展示工具名、参数、资源和预期影响，审批后冻结参数并执行，所有动作记录审计日志。关键业务动作可要求双人审批或禁止自动化。

### Q24：如何防 Prompt Injection？

**推荐回答：**

要区分系统指令、用户输入和外部内容的可信级别。RAG 文档、网页和工具返回只能作为不可信证据，不能改变系统规则。工具调用前必须由后端做权限和参数校验，高风险动作必须审批。上下文中要标注来源，并对外部内容做清洗和检测。

## 10. LangGraph 与框架类

### Q25：为什么选择 LangGraph？

**推荐回答：**

我选择 LangGraph 是因为它适合有状态、多步骤、可恢复、可插入人工审批的 Agent Workflow。它用 State、Node、Edge、Conditional Edge 和 Checkpoint 显式表达执行过程，适合把 RAG、工具调用、多 Agent、HITL 和监控整合成生产系统。相比只用 Chain，它更适合复杂流程；相比完全自治 Agent，它更可控。

### Q26：LangChain 和 LangGraph 有什么区别？

**推荐回答：**

LangChain 更偏 LLM 应用组件库，提供模型、Prompt、Retriever、Tool、Parser 等组件；LangGraph 更偏有状态 Agent Workflow 编排，用于管理节点、边、状态、条件路由、循环和 checkpoint。实际项目中可以用 LangChain 组件，用 LangGraph 做编排层。

### Q27：AutoGen、CrewAI、LangGraph 怎么选？

**推荐回答：**

LangGraph 适合有状态、可恢复、可治理的 Agent Workflow；AutoGen 和 CrewAI 更常被用于多 Agent 协作和角色对话原型。选型要看任务是否需要状态机、人工审批、持久化、生产监控和框架生态。面试中我会强调先掌握通用 Agent 编排思想，再选择框架。

## 11. 项目表达类

### Q28：如何设计一个端到端 Agent 项目？

**推荐回答：**

我会从业务目标开始，明确用户场景和成功指标；架构上用 FastAPI 接入，LangGraph 编排，RAG 提供知识，Tool Layer 调用外部系统，State 管理任务，Redis/PostgreSQL/Vector DB 做存储，评测监控和安全风控贯穿全链路。最后说明 Bad Case 如何回流，失败如何降级。

### Q29：如何体现项目不是 demo？

**推荐回答：**

要讲清生产化设计：权限、状态持久化、日志、trace、评测集、线上监控、重试、降级、HITL、审计和回滚。如果只说“用了模型 API 和向量库”，很容易被认为是 demo。

### Q30：面试中如何回答“你做过 Agent 吗”？

**推荐回答：**

可以按项目闭环回答：我做的是一个面向某业务场景的 Agent 系统，目标是解决什么问题。系统用 LangGraph 管理多步骤流程，用 RAG 获取知识，用工具调用连接外部系统，用 State 记录任务过程，用评测和监控保证质量。重点讲一个最难的问题，例如召回不准、工具调用不稳、状态恢复或安全审批，以及你如何解决。

## 12. 面试表达原则

```text
不要只说概念，要说工程边界；
不要只说用了框架，要说为什么选；
不要只说模型好，要说如何评测；
不要只说能自动化，要说如何控风险；
不要只说最终答案，要说过程 trace；
不要只说成功案例，要说失败怎么排查。
```
