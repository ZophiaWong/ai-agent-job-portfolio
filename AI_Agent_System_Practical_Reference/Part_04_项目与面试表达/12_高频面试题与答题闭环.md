# 12_高频面试题与答题闭环

## 0. 章节定位

本章用于 C02–C09 的冷答练习，也可用来追问 C10、C11 和 C13。题目在前，核对内容统一放在分隔线之后。

## 1. 冷答方法

1. 先选题，不向下滚动。用计时器口述，或把答案写入 `cold-answers.md`。
2. 回答时先澄清条件，再说机制、边界、失败模式和验证方法。系统设计题要留下图或文字数据流。
3. 作答后再展开对应内容，标出遗漏和错误。改完后做一道迁移题，并安排延迟复测。

阅读本页的答案不算能力证据。只保留作答前的记录、修正过程和延迟复测结果。

## 3. 参考答案与评分

<details>
<summary>Q1–Q3：Agent 基础</summary>

<a id="q1-answer"></a>

- Q1：Agent 是在明确边界内为任务目标选择步骤、调用工具、读取观察并更新状态的系统。LLM 可以负责局部判断，不等于整个系统。
- Q2：Chatbot 主要交付文本；Agent 还会触发工具和外部状态变化，所以需要权限、幂等、恢复、评测和审计。
- Q3：ReAct 交替进行判断、行动和观察。工程上记录结构化 Action/Observation，设置最大步数、终止条件和错误路由，不向用户暴露内部思维原文。

评分要点：是否说清系统边界、状态变化和停止条件。
</details>

<details>
<summary>Q4–Q6：Workflow 与多 Agent</summary>

- Q4：Chain 是固定线性步骤；Workflow 显式表达状态、分支、循环和错误处理；Agent 让模型在受限空间内动态选择下一步。
- Q5：任务有持久状态、条件路由、中断恢复或 HITL 时，状态机框架才有明确价值。简单请求不必为了框架而增加状态层。
- Q6：多 Agent 可以分工或并行，但会增加通信、状态一致性和成本问题。需要明确输入输出、共享状态、最大轮次和中止路径。

评分要点：是否给出选型条件，而不是只列框架概念。
</details>

<details>
<summary>Q7–Q10：RAG 工程</summary>

- Q7：离线链路处理解析、清洗、切块、元数据、embedding、建索引和版本；在线链路处理权限、query rewrite、召回、重排、上下文、生成和引用校验。
- Q8：先区分入库、召回、排序、上下文组装和生成失败。查 gold evidence 是否入库、是否进 TopK、排位如何、是否进入最终上下文。
- Q9：权衡语义完整性、召回粒度、噪声和成本。文档、表格和代码需要不同边界；用代表性查询评测，不靠固定 token 数猜。
- Q10：初召回要高召回率，但候选排序不足时再用 rerank。比较加入前后的 MRR/nDCG、上下文命中与端到端结果，同时记录延迟和成本。

评分要点：是否能从 trace 定位失败层，并提出可比较的验证。
</details>

<details>
<summary>Q11–Q15：上下文、记忆与微调</summary>

- Q11：Prompt Engineering 设计指令和输出约束；Context Engineering 决定哪些历史、证据、工具结果和状态进入模型，以及如何排序和淘汰。
- Q12：State 是当前任务可恢复的运行状态；Memory 是跨任务复用的信息。两者都要有 schema、权限、版本和删除策略。
- Q13：长期记忆先进候选区，再按稳定性、置信度、权限和可复用性写入。写入前去重与检查冲突，读取时做数据范围过滤。
- Q14：指令与格式用 Prompt，外部事实用 RAG，操作外部系统用工具，多步可控状态用 Workflow。这些边界稳定后，长期行为仍不符合要求才评估微调。
- Q15：LoRA 训练低秩增量参数；QLoRA 在量化基座模型上训练 adapter，显存需求更低。它们不适合保存频繁变动、需要引用的知识。

评分要点：是否用问题类型做决策，并说明数据、权限和更新边界。
</details>

<details>
<summary>Q16–Q18：工具与 MCP</summary>

- Q16：模型提议工具名和参数；后端做 schema、权限、策略、幂等和风险检查；执行器运行工具，将标准化 Observation 回填。执行权不在模型。
- Q17：区分只读和写操作，分类可重试错误，传递超时与取消，写操作使用幂等键。审批后冻结参数，保留不可变审计记录。
- Q18：Function Calling 是模型表达调用意图的机制；MCP 规范 host/client/server 间的工具与资源发现、schema 和传输。MCP 不取代应用侧权限与审计。

评分要点：是否把模型提议和真实执行分开，并处理副作用。
</details>

<details>
<summary>Q19–Q21：Python 后端与排障</summary>

- Q19：可以用 FastAPI/Pydantic 定义请求边界，Workflow 管理状态与路由，工具层隔离 RAG/API/SQL，PostgreSQL 保存任务与审计，Redis 只用于有明确生命周期的缓存、锁或短期状态。
- Q20：asyncio 处理单请求内可取消的 I/O 并发；任务队列处理跨请求长任务、延迟重试和 worker 负载分配。CPU 密集任务还需要进程或外部计算边界。
- Q21：先固定时间窗和版本变化，再按 Workflow、Tool、RAG、Context/Prompt 和 Model 分层。对比节点路由、错误码、召回文档、最终上下文与输出，用可复现 bad case 验证修复。

评分要点：是否说明取消、持久化和可复现排障，而不是只列技术栈。
</details>

<details>
<summary>Q22–Q24：安全与 HITL</summary>

- Q22：Chatbot 主要影响文本输出；Agent 可访问数据、执行代码和产生副作用，因此还要处理授权、数据范围、沙箱、幂等、HITL 和审计。
- Q23：模型只提议动作。后端生成待审批记录，展示参数和预期影响；审批后冻结参数，用幂等键执行，记录提议、审批、执行和结果。过期或参数改变要重新审批。
- Q24：系统指令、用户输入和外部内容要标明信任等级。RAG 文档和工具返回是不可信数据，不能改写系统规则；工具前置权限和参数检查仍由后端执行。

评分要点：是否说清信任边界、状态转移和重放安全。
</details>

<details>
<summary>Q25–Q27：框架选型</summary>

- Q25：LangGraph 适合有状态、多步、条件路由、中断恢复和 HITL 的任务。单次模型调用或固定短链路用普通函数更直接。
- Q26：LangChain 主要提供模型、prompt、retriever、tool 等组件；LangGraph 管理状态、节点、路由、循环和 checkpoint。实际系统可以同时使用，也可以都不使用。
- Q27：先问状态持久化、审批、恢复、多 Agent 通信、可观测性和团队维护成本，再用小型 spike 验证。框架名不是选型理由。

评分要点：是否给出不选某框架的条件与验证方法。
</details>

<details>
<summary>Q28–Q30：项目设计与事实边界</summary>

- Q28：先澄清用户、任务、成功标准、风险和规模，再画系统边界与数据流。说明状态、RAG、工具、存储、评测、降级和审计，最后用 bad case 检验设计。
- Q29：只引用可检查的源码、测试、trace、设计记录和运行结果。“用了某框架”不证明可靠性；要展示具体失败、修复和验证。
- Q30：先回答确实实现的范围，再给出证据路径。明确区分 implemented、planned、proposed improvement 和 unknown；通用架构练习不写成个人项目经历。

评分要点：是否先澄清约束，并能把每个实现主张指向证据。
</details>

## 4. 产物与复测

一次有效练习至少留下：

- 作答前的 `cold-answers.md`，包含用时、澄清问题和原始答案；
- 核对后的错误标注与一道迁移题；
- 隔几天重做的 `delayed-retest.md`，保留新答案而不覆盖旧记录。

要继续练系统设计，使用[Chapter 11](11_项目设计模板与架构表达.md)；要核对个人项目主张，使用[项目证据索引](../../projects/README.md)。
