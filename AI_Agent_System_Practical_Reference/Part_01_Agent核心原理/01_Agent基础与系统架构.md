## 1. 本章学习目标

本章的目标是建立 Agent 系统的第一性理解。学完后，你不应该只会说“Agent 会自主完成任务”，而应该能从系统角度解释：

```text
用户目标如何进入系统；
LLM 如何承担决策中枢；
Planner 如何拆解任务；
Tool 如何执行外部动作；
Observation 如何反馈给模型；
State 如何记录过程；
Memory 如何支撑跨轮任务；
安全、评测和监控如何保证系统可上线。
```

一个合格的 AI Agent 应用工程师，需要把 Agent 看成一个“可编排、可观测、可治理的任务执行系统”，而不是一个更会聊天的模型。

## 2. 核心概念总览

### 2.1 Agent 的一句话定义

LLM Agent 是以大语言模型为推理和决策核心，结合工具调用、状态管理、记忆系统、任务规划和反馈机制，在一定边界内自动或半自动完成任务的软件系统。

更工程化的表达是：

```text
Agent = LLM 决策能力
      + Workflow / Planning
      + Tool Use
      + State Management
      + Memory
      + Evaluation
      + Safety Guardrails
      + Observability
```

普通 Chatbot 的主要输出是文本，Agent 的目标是完成任务。它可能最终也输出文本，但中间会执行检索、调用 API、查询数据库、运行代码、发送通知、生成报告、触发审批等动作。

### 2.2 Agent 与 Chatbot 的区别

```text
Chatbot：
用户问题
  ↓
模型生成文本
  ↓
返回答案

Agent：
用户目标
  ↓
理解任务
  ↓
规划步骤
  ↓
选择工具
  ↓
执行动作
  ↓
观察结果
  ↓
更新状态
  ↓
继续规划或结束
  ↓
返回结果 / 触发业务动作
```

核心区别不在于模型能力，而在于系统边界：

| 维度 | 普通 Chatbot | AI Agent |
|---|---|---|
| 目标 | 回答问题 | 完成任务 |
| 动作 | 主要生成文本 | 可调用工具和外部系统 |
| 状态 | 多数是对话历史 | 任务状态、工具结果、计划、记忆 |
| 控制 | 一轮或短链路 | 多步骤、循环、分支、恢复 |
| 风险 | 错答、幻觉 | 错答 + 错操作 + 越权 + 副作用 |
| 工程重点 | Prompt、模型选择 | Workflow、工具、状态、评测、安全 |

## 3. 关键知识点详解
### 3.1 LLM 决策中枢

**是什么：** LLM 在 Agent 中不是单纯的文本生成器，而是负责理解目标、选择下一步动作、解释观察结果、决定是否继续执行的决策模块。

**为什么需要：** 开放式任务很难完全用规则写死。LLM 能基于自然语言目标、上下文和工具描述做动态决策，因此适合作为 Agent 的认知核心。

**核心机制：** LLM 接收系统提示词、用户目标、当前状态、可用工具、历史观察和约束规则，输出下一步计划、工具调用或最终答案。

**工程实现：** 工程上通常通过模型网关封装 LLM，统一处理模型路由、重试、超时、成本、日志和结构化输出。重要任务尽量使用 JSON Schema、function calling 或 Pydantic 校验输出。

**常见坑：**

- 把所有决策都交给模型，缺少边界控制。
- Prompt 中工具描述不清，导致工具误用。
- 没有结构化输出，后端无法稳定解析。
- 模型版本变化后行为漂移，没有回归测试。

**优化方式：**

- 为工具和动作定义明确 schema。
- 对关键决策使用规则或状态机兜底。
- 记录模型输入输出和路由原因。
- 对高风险动作引入人工审批。

**面试表达：** 我会把 LLM 看作 Agent 的决策中枢，但不会让它直接操作业务系统。模型只生成结构化意图和参数，真正执行由后端 Orchestrator 在权限、schema、幂等和审计约束下完成。
### 3.2 Planner / Executor

**是什么：** Planner 负责把用户目标拆成步骤，Executor 负责按步骤调用工具或执行动作。

**为什么需要：** 复杂任务通常无法一步完成。拆成计划和执行，可以降低单次推理难度，也方便插入检查、回滚、重试和人工确认。

**核心机制：** Planner 输出任务列表、依赖关系、成功条件和风险等级；Executor 按计划执行，并把每一步 Observation 写回状态。

**工程实现：** 在 LangGraph 中，Planner 可以是一个节点，Executor 可以是一组工具节点或子图。条件边根据执行结果决定继续、重试、转人工或结束。

**常见坑：**

- 计划过细导致步骤过多、成本高。
- 计划过粗导致执行节点缺少上下文。
- Planner 生成不可执行步骤。
- Executor 失败后没有恢复路径。

**优化方式：**

- 让 Planner 输出可执行动作，而不是抽象愿望。
- 为每步定义完成条件。
- 限制最大步数和重试次数。
- 在执行后加入 Critic 或 Validator 节点。

**面试表达：** Planner-Executor 模式的价值是把“想做什么”和“怎么执行”分开，便于控制复杂任务的步骤、状态和风险。
### 3.3 ReAct：Reasoning + Acting

**是什么：** ReAct 是让模型在推理和行动之间交替进行的 Agent 思想，即思考下一步、执行动作、观察结果，再继续思考。

**为什么需要：** 单纯推理容易脱离环境，单纯行动容易盲目。ReAct 通过 Observation 把外部反馈引入推理过程，让 Agent 能动态修正。

**核心机制：** 典型循环是 Thought → Action → Observation → Thought。工程实现时不一定暴露 Thought，但一定要保留 Action 和 Observation 的结构化日志。

**工程实现：** 可以将每次工具调用作为 Action，将工具返回作为 Observation，写入 State。下一轮模型基于 Observation 决定继续、修改计划或终止。

**常见坑：**

- 无限循环。
- Observation 太长导致上下文污染。
- 工具返回错误但模型误信。
- 把内部推理原文直接暴露给用户。

**优化方式：**

- 设置最大迭代次数。
- 对 Observation 做摘要和结构化。
- 工具结果增加置信度和错误码。
- 对最终答案做证据一致性检查。

**面试表达：** ReAct 的核心不是展示模型思考过程，而是建立“行动—反馈—修正”的闭环。生产系统要把这个闭环变成可记录、可评测、可中断的执行链路。
### 3.4 Memory 与 State

**是什么：** State 是当前任务执行中的运行状态，Memory 是跨步骤或跨会话可复用的信息。

**为什么需要：** Agent 不是单轮问答，必须知道已经做了什么、下一步要做什么、工具返回了什么、用户偏好是什么。

**核心机制：** State 通常记录任务 ID、当前步骤、计划、工具结果、错误状态、审批状态等；Memory 可分为短期任务记忆和长期用户/业务记忆。

**工程实现：** 在 Python 系统中，短期 State 可由 LangGraph State、Redis、数据库任务表承载；长期 Memory 可由向量库、PostgreSQL、对象存储或图数据库承载。

**常见坑：**

- 把所有历史都塞进上下文。
- 长期记忆无条件写入导致污染。
- State 字段过多，节点耦合严重。
- 缺少状态版本，失败后无法恢复。

**优化方式：**

- State 只保留决策必需字段。
- 原始证据外部存储，State 保存引用 ID。
- 长期记忆写入前做去重、置信度和生命周期判断。
- 使用 checkpoint 支持恢复。

**面试表达：** State 解决当前任务进行到哪一步，Memory 解决跨任务可以复用什么。二者混在一起会导致上下文膨胀和记忆污染。

## 5. 工程实战设计

### 5.1 一个 Agent 请求的生命周期

```text
1. 用户提交目标；
2. API 层创建 task_id 和 trace_id；
3. Orchestrator 构造初始 State；
4. LLM 判断任务类型和风险等级；
5. Planner 生成执行计划；
6. Workflow 根据条件路由到检索、工具、人工审批或答案生成；
7. 每一步输出写回 State；
8. 工具返回进入 Observation；
9. Validator 检查是否满足完成条件；
10. 成功则生成最终结果，失败则重试、降级或转人工；
11. 全链路写日志、指标、Trace 和审计。
```

### 5.2 生产化 Agent 的最小能力集

一个可上线 Agent 至少需要：

```text
结构化输入输出；
工具 schema；
状态持久化；
最大步数和超时；
重试和降级；
权限校验；
高风险动作审批；
节点级日志；
成本和延迟监控；
Bad Case 回流。
```

## 7. 常见误区

### 7.1 把 Agent 等同于“提示词写得更长”

Prompt 很重要，但 Agent 的核心是行动闭环和状态治理。只靠长 Prompt 无法解决工具失败、状态恢复、权限控制和审计问题。

### 7.2 把 Agent 做成完全自治

生产环境通常不应该一开始追求完全自治。更稳妥的路线是：

```text
人工辅助
  ↓
半自动执行
  ↓
低风险自动化
  ↓
高风险动作审批后执行
  ↓
逐步扩大自动化范围
```

### 7.3 忽视失败恢复

Agent 多步骤执行时失败是常态。真正的工程能力体现在：失败能否定位、重试、回滚、降级和复盘。

## 8. 面试题与答题闭环
### Q：什么是 AI Agent？

**考察点：** 定义、与 Chatbot 区别、系统组件、工程边界。

**推荐回答：** AI Agent 是以 LLM 为决策核心，结合规划、工具调用、状态管理、记忆和反馈机制完成任务的软件系统。回答时我会先区分它和 Chatbot：Chatbot 主要生成文本，Agent 需要在多步骤流程中选择工具、执行动作、观察结果并修正计划。工程上我会强调 Orchestrator、State、Tool Layer、Evaluation 和 Safety，而不是只说模型更智能。

### Q：Agent 为什么难以上线？

**考察点：** 生产化风险、失败模式、治理能力。

**推荐回答：** 因为上线难点不只是模型回答质量，而是长链路稳定性、工具可靠性、状态一致性、权限安全、成本延迟和可观测性。一次 Agent 任务可能包含多次模型调用和多次工具调用，任何一个节点失败都会影响最终任务成功率，所以需要状态机、重试、熔断、降级、HITL 和 Bad Case 回流。

### Q：ReAct 的核心价值是什么？

**考察点：** 推理-行动闭环、Observation、工程实现。

**推荐回答：** ReAct 的价值是让模型在推理和行动之间形成闭环：先判断下一步，再调用工具，读取 Observation 后继续修正。生产实现时我不会把内部思考裸露给用户，而会记录结构化 Action、Observation 和状态变更，用于调试、评测和审计。
