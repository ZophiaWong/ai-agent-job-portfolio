## 1. 本章学习目标

工具调用是 Agent 从“回答问题”走向“执行任务”的关键。没有工具，Agent 只能生成文本；有了工具，Agent 可以检索知识、查数据库、调用业务 API、执行代码、读写文件、创建工单或触发审批。

但工具调用也是 Agent 风险最大的部分。生产系统必须把模型的“想调用工具”变成后端可控的结构化请求：

```text
模型提出工具调用意图
  ↓
后端校验工具名、参数、权限、风险等级
  ↓
执行或拒绝
  ↓
记录日志和审计
  ↓
将标准化结果回填模型
```

## 2. 核心概念总览

### 2.1 Function Calling

Function Calling 是让模型输出结构化函数调用意图的一类机制。模型不直接执行函数，而是输出：

```text
工具名：search_docs
参数：{"query": "Agent RAG 评测", "top_k": 5}
```

真正执行由后端完成。这个边界非常重要：

```text
模型负责选择和填参；
后端负责校验和执行；
业务系统负责权限和副作用控制。
```

### 2.2 Tool Schema

Tool Schema 描述工具的名称、用途、参数、类型、必填字段、枚举值和返回格式。好的 schema 能显著提升工具调用稳定性。

一个工具 schema 至少要说清：

```text
工具能做什么；
什么时候应该用；
什么时候不应该用；
参数有哪些；
参数类型和范围；
返回结果结构；
错误时返回什么；
是否有副作用；
风险等级是什么。
```

### 2.3 MCP 的定位

MCP 可以理解为模型和外部工具/资源之间的标准化协议层。它不是简单“多一个 API”，而是让工具发现、调用、参数 schema、资源暴露、权限边界和客户端交互更加规范。

在 Agent 工程中，可以这样理解：

```text
Function Calling：模型如何表达“我要调用哪个工具”
MCP：工具和资源如何被标准化暴露给模型/客户端
Orchestrator：谁负责把调用变成可控执行
```

## 3. 关键知识点详解
### 3.1 工具调用链路

**是什么：** 工具调用链路是从用户目标到模型生成工具调用，再到后端执行并回填结果的完整过程。

**为什么需要：** Agent 如果要解决真实任务，就必须连接外部系统。但模型不能直接获得无限权限，必须通过后端进行控制。

**核心机制：** 模型基于工具描述决定是否调用工具，并输出工具名和参数；后端做 schema 校验、权限校验、幂等检查、执行、错误处理和结果标准化。

**工程实现：** Python 中通常把工具定义成函数或类，配合 Pydantic schema 校验参数；在 LangGraph 中，工具调用可以放在 Tool Node 或自定义节点中。

**常见坑：**

- 工具描述模糊导致误调用。
- 参数没有校验，业务系统收到脏数据。
- 工具返回原始大文本，污染上下文。
- 写操作重复执行造成副作用。

**优化方式：**

- 写清工具用途和禁用场景。
- 使用 Pydantic / JSON Schema 校验。
- 工具结果结构化并摘要。
- 写操作引入幂等键和审批。

**面试表达：** 我会把工具调用拆成模型决策和后端执行两个阶段。模型只输出结构化调用意图，后端负责校验、权限、执行、重试、审计和回填。
### 3.2 工具可靠性治理

**是什么：** 工具可靠性治理是对工具调用过程中的超时、失败、限流、错误参数、重复调用和异常返回进行控制。

**为什么需要：** Agent 链路中工具失败是常态。如果没有治理，一个工具 429、超时或返回异常就会导致整个任务失败。

**核心机制：** 治理机制包括：超时、重试、熔断、限流、fallback、错误码标准化、参数校验、结果校验、幂等键、审计日志。

**工程实现：** 工程实现中，每个工具都应有统一 wrapper，负责记录 trace_id、latency、status_code、retry_count、error_type 和 result_summary。

**常见坑：**

- 所有错误都重试。
- 不可重试错误被重复调用。
- 写操作没有幂等。
- 工具失败后模型继续编造答案。

**优化方式：**

- 区分临时错误、参数错误、权限错误、业务错误。
- 只对可重试错误重试。
- 写操作必须使用 idempotency key。
- 工具失败时给模型明确错误状态。

**面试表达：** 工具可靠性不是让模型更聪明，而是让后端工具层像普通生产服务一样具备超时、重试、熔断、降级、日志和审计。
### 3.3 副作用操作与幂等

**是什么：** 副作用操作是会改变外部状态的动作，例如写数据库、发送邮件、创建订单、删除文件、转账、发布内容。

**为什么需要：** Agent 一旦能执行副作用操作，风险就从“回答错误”变成“操作错误”。重复调用或误调用可能造成真实损失。

**核心机制：** 幂等是指同一个业务请求重复执行多次，结果只生效一次。副作用工具必须支持幂等键、状态检查和审批机制。

**工程实现：** 后端可以基于 task_id + tool_name + business_key 生成幂等键，执行前检查是否已成功执行，执行后记录结果和审计。

**常见坑：**

- 重试导致重复发邮件。
- 同一任务并发调用同一写工具。
- 模型误判用户意图执行删除。
- 审批通过内容和最终执行内容不一致。

**优化方式：**

- 所有写工具要求幂等键。
- 高风险动作前展示执行计划和参数。
- 审批后冻结执行参数。
- 用分布式锁或数据库唯一约束防重复。

**面试表达：** 我会把工具分成只读工具和写工具。只读工具重点是时延和准确性，写工具重点是权限、幂等、审批、审计和回滚。
### 3.4 MCP 化工具

**是什么：** MCP 化工具是把普通 API、数据库查询、文件系统或业务能力包装成符合 MCP 约定的工具/资源，让 Agent 能以标准方式发现和调用。

**为什么需要：** 当工具来源多、模型/客户端多、团队多时，如果每个工具都自定义协议，会导致接入成本高、权限不统一、审计困难。

**核心机制：** MCP Server 暴露工具清单、schema 和调用接口；Client/Host 决定是否展示、确认和调用工具；工具端执行真实逻辑。

**工程实现：** 工程实现时，普通 API MCP 化要做包装、schema、权限、输入校验、输出标准化、错误码、超时、日志、契约测试。

**常见坑：**

- 把 MCP 当成绕过权限的万能工具层。
- 工具描述过宽，模型滥用。
- 缺少用户确认。
- 输出未经清洗直接回填模型。

**优化方式：**

- 工具最小权限暴露。
- 敏感工具必须确认。
- 工具输出做脱敏和结构化。
- 记录每次调用的用户、参数、结果和风险等级。

**面试表达：** MCP 的价值不是让模型能调更多工具，而是让工具暴露、发现、调用和治理更加标准化。

## 5. 工程实战设计

### 5.1 工具设计模板

```text
工具名：search_knowledge_base
工具类型：只读
适用场景：用户问题需要查询企业知识库
不适用场景：需要查询实时订单、用户隐私数据、外部网页
输入参数：
  - query: str，必填
  - top_k: int，默认 5，范围 1-20
  - filters: dict，可选
输出：
  - documents: list
  - citations: list
  - confidence: float
错误码：
  - EMPTY_QUERY
  - NO_RESULT
  - TIMEOUT
  - PERMISSION_DENIED
风险等级：低
是否需要审批：否
```

### 5.2 工具错误处理策略

```text
临时网络错误：
  可重试，设置最大次数和退避。

429 限流：
  可重试或切换备用工具，必要时降级。

参数非法：
  不重试，返回参数错误给模型修正。

权限错误：
  不重试，转人工或拒绝。

业务规则不允许：
  不重试，解释原因。

写操作失败：
  检查幂等记录，避免重复执行。
```

## 6. 与 LangGraph / Python 后端的映射

```text
工具概念                 LangGraph / Python 实现
------------------------------------------------------
工具函数                 Python function / class method
参数模型                 Pydantic BaseModel
工具节点                 Tool Node / 自定义 Node
工具结果                 Observation 写入 State
错误状态                 State.error / State.status
重试策略                 节点 wrapper / tenacity / 自定义 retry
高风险审批               Interrupt / Human Review Node
幂等记录                 PostgreSQL / Redis / 唯一索引
审计日志                 audit_log table
```

一个简化调用流程：

```text
FastAPI endpoint
  ↓
graph.stream(input_state)
  ↓
router_node 判断需要工具
  ↓
tool_node 执行工具 wrapper
  ↓
wrapper 做 schema + auth + retry + audit
  ↓
result 写入 state["observations"]
  ↓
conditional_edge 判断继续或结束
```

## 7. 常见误区

### 7.1 认为 Function Calling 会自动保证正确

Function Calling 只能让模型更容易输出结构化调用，不保证调用正确、参数合法或业务安全。真正的可靠性来自后端校验、权限、状态机和评测。

### 7.2 工具越多越好

工具越多，模型选择空间越大，误调用风险越高。工具应该按任务域聚合，描述清晰，必要时通过 Router 限制可见工具集合。

### 7.3 只关注成功调用，不关注失败语义

工具失败时，返回给模型的错误必须可解释。例如“权限不足”“参数缺失”“服务限流”“没有结果”是完全不同的后续路径。

## 8. 面试题与答题闭环
### Q：Function Calling 的完整流程是什么？

**考察点：** 结构化调用、后端执行边界、工具结果回填。

**推荐回答：** 模型根据工具描述输出工具名和参数，后端解析后做 schema 校验、权限校验、幂等检查和风险判断，再执行真实工具。工具结果标准化后作为 Observation 回填给模型，模型决定继续调用、修正或输出最终答案。

### Q：如何保证工具调用可靠？

**考察点：** 工具治理、生产可靠性。

**推荐回答：** 我会从工具 schema、参数校验、超时、重试、熔断、限流、fallback、幂等、错误码标准化、trace 和审计几个方面保证。尤其要区分只读工具和写工具，写工具必须支持幂等和人工审批。

### Q：MCP 和 Function Calling 有什么关系？

**考察点：** 协议定位、工具生态、工程边界。

**推荐回答：** Function Calling 更偏模型如何表达调用意图，MCP 更偏工具和资源如何标准化暴露、发现和调用。实际系统中，模型可以通过 function calling 选择工具，而工具集合可以由 MCP Server 提供，最终执行仍由后端在权限和审计约束下完成。
