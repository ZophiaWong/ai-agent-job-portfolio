## 1. 本章学习目标

Agent 安全比普通 Chatbot 更复杂，因为 Agent 可能调用工具并改变外部世界。错误回答是质量问题，错误操作可能是安全事故。

本章目标是建立分层安全观：

```text
输入安全
  ↓
身份认证
  ↓
权限判断
  ↓
工具白名单
  ↓
参数校验
  ↓
风险分级
  ↓
Human-in-the-loop
  ↓
沙箱执行
  ↓
输出审查
  ↓
审计日志
```

## 2. 核心概念总览

### 2.1 Agent 安全风险分类

```text
输入攻击：
  Prompt Injection、Jailbreak、恶意文档、恶意网页内容。

权限风险：
  越权读取数据、越权调用工具、跨用户数据泄露。

工具风险：
  错误调用、参数注入、重复执行、副作用不可控。

数据风险：
  敏感信息泄露、记忆污染、RAG 私有文档泄露。

执行风险：
  代码执行、文件删除、网络访问、生产环境修改。

输出风险：
  有害内容、错误建议、泄露内部信息、伪造引用。
```

### 2.2 安全不是只靠模型拒答

模型安全对齐只能解决一部分问题。生产 Agent 需要系统级控制：

```text
模型层：安全对齐、拒答策略
系统层：最小权限、工具白名单
执行层：沙箱、资源限制、网络隔离
业务层：风险分级、人工审批
观测层：审计、告警、回放
```

## 3. 关键知识点详解
### 3.1 Prompt Injection

**是什么：** Prompt Injection 是攻击者通过用户输入、文档内容或网页内容诱导模型忽略原有指令、泄露信息或误调用工具。

**为什么需要：** Agent 会读取外部内容，这些内容可能包含恶意指令。如果模型把外部内容当成系统指令，就可能被劫持。

**核心机制：** 常见攻击包括：要求忽略系统提示、要求输出密钥、诱导调用工具、在 RAG 文档中嵌入恶意指令。

**工程实现：** 工程上要区分系统指令、用户输入和外部内容的可信级别。RAG 文档应作为不可信证据，不可直接覆盖系统策略。

**常见坑：**

- 把检索文档原样放入上下文。
- 让模型自行判断权限。
- 外部网页内容可以指挥工具。
- Prompt 中没有说明外部内容不可信。

**优化方式：**

- 上下文分区标注来源。
- 系统策略优先级最高。
- 工具调用前做服务端权限校验。
- 对外部内容做 injection 检测和清洗。

**面试表达：** Prompt Injection 的关键防线不是让模型更听话，而是让外部内容没有权限修改系统规则或触发高风险工具。
### 3.2 权限与最小权限

**是什么：** 最小权限是指 Agent 只能访问完成当前任务所需的最少数据和工具。

**为什么需要：** Agent 如果拥有过宽权限，一次误判或攻击就可能泄露大量数据或执行危险动作。

**核心机制：** 权限控制应基于用户身份、会话、任务类型、资源归属、工具风险等级和业务策略。

**工程实现：** 后端必须做二次鉴权。模型不能决定用户是否有权限访问某个文档、数据库或工具。

**常见坑：**

- 把工具 token 放进 Prompt。
- 所有用户共用同一工具权限。
- RAG 检索后再过滤权限。
- 模型说可以就执行。

**优化方式：**

- 用户级凭证或服务端代理。
- 检索前权限过滤。
- 工具调用前资源归属校验。
- 敏感工具按任务动态授权。

**面试表达：** Agent 权限必须由后端系统控制，模型只提出意图，不能成为权限判断主体。
### 3.3 Human-in-the-loop

**是什么：** Human-in-the-loop 是在关键节点暂停 Agent，展示计划、参数和风险，等待人工审批后继续执行。

**为什么需要：** 高风险动作不能完全自动化，尤其是删除、发送、下单、交易、生产变更、外部发布等动作。

**核心机制：** HITL 可以分为审批、编辑、确认、驳回和转人工处理。审批结果应写回 State，并记录审计日志。

**工程实现：** 在 LangGraph 中可以通过 interrupt 或人工审批节点暂停工作流，审批后从 checkpoint 恢复执行。

**常见坑：**

- 审批内容和最终执行内容不一致。
- 只让用户点确认但不展示参数。
- 审批记录不完整。
- 审批后 Agent 又修改动作。

**优化方式：**

- 审批前冻结执行参数。
- 展示风险、工具名、资源和预期影响。
- 记录审批人、时间、内容和结果。
- 审批后只允许执行已批准参数。

**面试表达：** HITL 不是降低智能，而是把高风险动作从自动决策变成受控执行。
### 3.4 沙箱与执行隔离

**是什么：** 沙箱是限制 Agent 执行环境的隔离机制，例如限制文件系统、网络、CPU、内存、时间和可访问资源。

**为什么需要：** 代码执行、浏览器操作和文件处理都可能带来安全风险。沙箱能降低恶意代码或错误动作的影响范围。

**核心机制：** 沙箱可以是 Docker、VM、受限容器、云执行环境或浏览器隔离环境。

**工程实现：** 工程上需要限制可写目录、网络出口、运行时长、资源配额，并记录执行日志和产物。

**常见坑：**

- 让 Agent 在生产主机直接执行代码。
- 沙箱有外网无限访问。
- 没有资源限制。
- 执行产物不审查。

**优化方式：**

- 容器隔离。
- 网络白名单。
- 只读挂载敏感文件。
- CPU/内存/时间限制。
- 执行结果扫描和审计。

**面试表达：** 能执行代码的 Agent 必须默认不信任，沙箱是基础设施级安全边界。
### 3.5 红队测试与安全评测

**是什么：** 红队测试是主动模拟攻击者，测试 Agent 是否会越权、泄露、误操作或绕过策略。

**为什么需要：** Agent 安全问题往往只在恶意输入、复杂上下文和工具组合下暴露。

**核心机制：** 测试样本应覆盖 prompt injection、jailbreak、权限越界、敏感信息提取、工具滥用、重复副作用、恶意 RAG 文档。

**工程实现：** 红队结果应回流到策略、工具权限、Prompt、评测集和监控告警。

**常见坑：**

- 只做普通问答测试。
- 没有工具层安全测试。
- 红队样本不上线回归。
- 发现问题只改 Prompt。

**优化方式：**

- 建立安全评测集。
- 每次版本发布跑安全回归。
- 高风险工具做专项测试。
- 将安全事件加入告警和审计。

**面试表达：** 安全评测要覆盖输入、上下文、工具、权限、输出，而不是只测模型拒答。

## 5. 工程实战设计

### 5.1 工具风险分级

```text
L0：纯文本处理
  摘要、改写、分类，无外部访问。

L1：只读查询
  RAG 检索、查询公开数据、读取用户有权限文档。

L2：低风险写操作
  创建草稿、生成报告、保存临时结果。

L3：高风险写操作
  发送邮件、提交工单、修改数据库、删除文件。

L4：关键业务动作
  金融交易、生产变更、下单、对外发布、权限修改。
```

治理建议：

```text
L0/L1：自动执行，记录日志；
L2：可自动执行，但需幂等和可撤销；
L3：必须展示参数并人工确认；
L4：双人审批、强审计、可回滚或禁止自动执行。
```

## 7. 常见误区

### 7.1 认为系统提示词可以保证安全

系统提示词有帮助，但不能替代权限、沙箱、审批和审计。攻击者可以通过外部内容诱导模型，但不能绕过后端权限校验。

### 7.2 高风险动作只靠“用户确认”

确认必须展示具体参数和影响，并保证审批后的执行参数不被模型修改。

### 7.3 工具权限过宽

工具越强，权限越要细。最好按用户、资源、任务动态授权，而不是给 Agent 一个万能 token。

## 8. 面试题与答题闭环
### Q：Agent 安全和普通 Chatbot 安全有什么不同？

**考察点：** 安全范围差异。

**推荐回答：** 普通 Chatbot 主要风险是有害输出和幻觉；Agent 还会调用工具、读写数据、执行代码和触发业务动作，所以风险包括越权、数据泄露、工具滥用、副作用重复执行和生产变更。治理必须包括权限、工具白名单、沙箱、HITL、审计和监控。

### Q：如何治理高风险工具调用？

**考察点：** 风险分级、HITL、审计。

**推荐回答：** 我会按风险分级。只读工具可自动执行，写工具必须幂等，高风险动作要人工审批，审批前展示工具名、参数、资源、预期影响，审批后冻结参数并记录审计。关键业务动作可以要求双人审批或禁止自动执行。

### Q：如何防 Prompt Injection？

**考察点：** 提示注入与系统边界。

**推荐回答：** 要区分系统指令、用户输入和外部内容可信级别；外部文档只能作为证据，不能改变系统规则。工具调用前必须服务端校验权限和参数，RAG 内容要标注来源并做清洗，高风险工具必须审批。
