## 1. 本章学习目标

本章解决一个高频工程问题：Agent 效果不好时，到底应该改 Prompt、改上下文、改 RAG、改工具链路，还是做微调？

很多项目失败不是因为不会某一种技术，而是优化方向选错。例如：

```text
知识缺失却去微调；
工具 schema 混乱却改 prompt；
上下文污染却换模型；
没有评测集却盲目调参数；
格式不稳定却做向量库优化。
```

学完本章后，你需要形成一套判断框架。

## 2. 核心概念总览

### 2.1 四层优化顺序

```text
第一层：Prompt / Structured Output
  解决输出格式、角色、边界、语气、约束问题。

第二层：Context / RAG / Tool Chain
  解决知识、证据、工具结果、上下文组织和状态问题。

第三层：Workflow / Guardrails / Evaluation
  解决流程可控、质量门禁、错误恢复和评测回流问题。

第四层：SFT / LoRA / DPO 等微调
  解决长期行为模式、风格、格式、领域表达和偏好对齐问题。
```

### 2.2 判断框架

```text
问题是格式不稳定？
  → Prompt + JSON Schema + Output Parser

问题是事实错误或知识过期？
  → RAG + 数据治理 + 引用校验

问题是检索到了但没用好？
  → Context Engineering + Rerank + 答案约束

问题是工具调用不稳定？
  → Tool Schema + Few-shot Tool Trace + Workflow

问题是流程循环或乱跳？
  → 状态机 + 终止条件 + 评测

问题是长期风格/行为不一致？
  → 微调或偏好优化
```

## 3. 关键知识点详解
### 3.1 Prompt 优化

**是什么：** Prompt 优化是通过角色、任务、约束、示例、输出格式和边界条件改善模型输出。

**为什么需要：** 很多基础问题，如输出格式不稳定、回答风格不一致、拒答边界不清，可以先通过 Prompt 解决。

**核心机制：** 高质量 Prompt 通常包含任务目标、角色、输入说明、输出格式、约束、示例、禁止行为和失败处理。

**工程实现：** 工程上应把 Prompt 版本化，配合评测集做 A/B 测试。结构化输出尽量使用 JSON Schema、Pydantic 或函数调用，而不是纯自然语言约束。

**常见坑：**

- Prompt 越写越长但冲突越来越多。
- 没有版本管理。
- 示例和规则不一致。
- 用 Prompt 解决本该由工具或权限解决的问题。

**优化方式：**

- Prompt 模块化。
- 明确优先级。
- 加入正例和反例。
- 对输出做解析和校验。
- 建立 Prompt 回归测试。

**面试表达：** Prompt 是低成本优化手段，但它不能替代 RAG、工具治理、权限和评测。面试中应强调先做低成本可验证优化。
### 3.2 Context 优化

**是什么：** Context 优化是选择、排序、压缩和注入模型输入信息的过程。

**为什么需要：** 同一个 Prompt，在不同上下文下表现完全不同。上下文噪声、缺失、冲突或顺序不当都会影响结果。

**核心机制：** Context 优化包括历史摘要、RAG 证据筛选、工具结果摘要、状态裁剪、长期记忆读取和 token budget。

**工程实现：** 工程上可以建立 context_builder，把每类信息按优先级、预算和来源打包，并记录本轮最终输入，便于调试。

**常见坑：**

- 把所有信息塞进去。
- 上下文来源不可追溯。
- 低相关记忆污染任务。
- 关键信息被放在中间或截断。

**优化方式：**

- 设置 context budget。
- 按任务阶段选择信息。
- 保留引用 ID。
- 对上下文构造结果做日志和评测。

**面试表达：** 当模型答案混乱时，不要只改 Prompt，先检查上下文是否干净、相关、完整和可追溯。
### 3.3 RAG 优化

**是什么：** RAG 优化是提升外部知识召回、排序、上下文组装和引用质量。

**为什么需要：** 事实错误、领域知识缺失、私有数据问答和知识更新问题，通常优先考虑 RAG 而不是微调。

**核心机制：** RAG 优化包括数据清洗、chunking、embedding、hybrid search、rerank、query rewrite、context compression 和 citation validation。

**工程实现：** 工程上需要分离检索评测和生成评测，先判断证据是否召回，再判断模型是否正确使用证据。

**常见坑：**

- 把 RAG 效果差归因于模型。
- 只调 top_k。
- 忽视文档质量和权限。
- 没有检索评测集。

**优化方式：**

- 构建 query-gold-doc 数据集。
- 用 Recall@K、MRR、nDCG 评估检索。
- 用人工或 LLM judge 评估答案引用。
- Bad Case 回流到数据和检索链路。

**面试表达：** 知识问题优先 RAG，行为问题再考虑微调。RAG 的核心是证据链路，不是向量库本身。
### 3.4 工具链路优化

**是什么：** 工具链路优化是改善 Agent 选择工具、填充参数、处理错误和吸收结果的能力。

**为什么需要：** 工具调用失败通常不是模型能力问题，而是工具描述、参数 schema、错误语义、工作流和权限设计问题。

**核心机制：** 优化方向包括工具描述、参数类型、示例调用、错误码、重试策略、工具结果摘要和观察结果注入方式。

**工程实现：** 在 LangGraph 中可以通过 Tool Node、Router、Validator 和 Error Handler 节点控制工具调用路径。

**常见坑：**

- 工具太多导致选择混乱。
- schema 过宽。
- 错误返回自然语言且不可解析。
- 模型不知道工具失败后怎么办。

**优化方式：**

- 按任务域缩小工具集合。
- 明确工具禁用场景。
- 错误结果结构化。
- 为工具调用提供 few-shot trace。

**面试表达：** 工具调用不稳时，应先改工具 schema 和工作流，而不是立刻做模型微调。
### 3.5 微调触发条件

**是什么：** 微调是通过训练数据修改模型行为分布，使其更稳定地遵循某类格式、风格、任务模式或偏好。

**为什么需要：** 当 Prompt、Context、RAG、工具链路和 Workflow 已优化，但模型仍长期在某类行为上不稳定，才考虑微调。

**核心机制：** 微调适合输出风格、格式、领域表达、工具调用轨迹、特定任务模式；不适合频繁变化的事实知识。

**工程实现：** 工程上需要高质量训练集、验证集、离线评测、安全评测、灰度发布和回滚策略。

**常见坑：**

- 用微调补私有知识。
- 数据质量差。
- 没有基线和评测。
- 微调后破坏通用能力。

**优化方式：**

- 先建立评测集。
- 收集成功轨迹和修正样本。
- 小规模验证。
- 上线前做回归和安全测试。

**面试表达：** 我会把微调作为最后的行为稳定化手段，而不是一开始解决所有问题的银弹。

## 5. 工程实战设计

### 5.1 优化闭环

```text
收集 Bad Case
  ↓
分类：
  - 检索失败
  - 上下文污染
  - 工具失败
  - 格式错误
  - 推理错误
  - 安全问题
  ↓
定位责任层：
  - Prompt
  - Context
  - RAG
  - Tool
  - Workflow
  - Model
  ↓
制定最小修改
  ↓
离线评测
  ↓
小流量灰度
  ↓
线上监控
  ↓
回流新 Bad Case
```

### 5.2 改动优先级原则

```text
优先改低成本、可回滚、可评测的部分；
不要在没有评测集时大改 Prompt 或换模型；
知识问题优先改数据和检索；
工具问题优先改 schema 和错误处理；
行为稳定性问题再考虑微调。
```

## 7. 常见误区

### 7.1 效果差就换模型

模型升级可能改善结果，但也可能引入行为漂移。应先定位问题层级，避免用昂贵方法掩盖工程问题。

### 7.2 微调优先于 RAG

事实性、时效性、私有知识通常优先 RAG。微调更适合稳定行为模式和表达风格。

### 7.3 没有评测就优化

没有评测集的优化只能靠感觉，很容易局部变好、整体变差。

## 8. 面试题与答题闭环
### Q：什么时候用 Prompt，什么时候用 RAG，什么时候微调？

**考察点：** 优化路线判断。

**推荐回答：** 输出格式、语气、边界和简单约束问题优先 Prompt；事实知识、私有数据、时效性和可追溯性问题优先 RAG；当 Prompt、RAG、工具和 Workflow 都优化后，模型在长期行为、格式或偏好上仍不稳定，再考虑 SFT、LoRA 或 DPO。

### Q：Agent 效果不好怎么排查？

**考察点：** 分层排查和工程优化。

**推荐回答：** 先做 Bad Case 分类：格式错、知识错、检索错、工具错、流程错、安全错。然后按 Prompt、Context、RAG、Tool、Workflow、Model 分层定位。不要一上来换模型或微调，要先找最小可验证改动。

### Q：工具调用不稳定是否应该微调？

**考察点：** 工具链路与微调边界。

**推荐回答：** 不一定。应该先检查工具描述、参数 schema、示例、错误返回、可见工具集合和工作流路由。如果工具链路仍然稳定性不足，并且有大量高质量工具调用轨迹，再考虑轨迹微调。
