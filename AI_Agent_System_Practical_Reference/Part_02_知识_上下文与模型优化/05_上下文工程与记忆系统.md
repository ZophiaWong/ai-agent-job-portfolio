上下文工程是 Agent 应用的核心能力之一。模型表现好不好，不只取决于模型本身，还取决于你给它什么信息、按什么顺序给、给多少、什么时候更新、什么时候删除。

## 2. 核心概念总览

### 2.1 Prompt Engineering vs Context Engineering

```text
Prompt Engineering：
关注如何写指令、角色、格式、约束、示例。

Context Engineering：
关注哪些信息进入模型、如何选择、排序、压缩、更新、淘汰和回取。
```

Prompt 是模型输入的一部分，Context Engineering 是围绕模型输入进行的信息治理系统。

### 2.2 Agent 上下文来源

```text
用户当前输入
  +
系统提示词
  +
开发者规则
  +
历史对话摘要
  +
当前任务状态
  +
工具调用结果
  +
RAG 检索证据
  +
长期用户记忆
  +
业务权限与策略
  +
错误和重试历史
  ↓
上下文选择 / 排序 / 压缩
  ↓
模型输入
```

### 2.3 State、Memory、Context 的区别

```text
State：
当前任务运行状态，例如步骤、计划、工具结果、错误、审批状态。

Memory：
跨步骤或跨会话可复用的信息，例如用户偏好、历史经验、长期事实。

Context：
本次模型调用实际看到的输入信息，来自 State、Memory、RAG、Prompt 等。
```

## 3. 关键知识点详解
### 3.1 短期记忆

**是什么：** 短期记忆是当前会话或当前任务内需要保留的信息，例如用户目标、执行计划、中间结果、工具返回、错误记录。

**为什么需要：** Agent 多步骤执行时必须知道已经做了什么、下一步要做什么，否则会重复执行或偏离目标。

**核心机制：** 短期记忆通常与 State 结合，任务结束后可以归档或摘要，不一定进入长期记忆。

**工程实现：** Python 中可以把短期状态放在 LangGraph State、Redis、数据库任务表或 checkpoint 中。长文本结果不建议直接放 State，可放对象存储并保存引用。

**常见坑：**

- 把完整历史无差别塞给模型。
- 工具结果太长导致上下文超限。
- 状态字段不结构化，节点难以读取。
- 任务结束后短期状态没有归档。

**优化方式：**

- 把历史做阶段摘要。
- 工具结果结构化。
- State 字段按职责拆分。
- 大文本外部存储，State 保存 ID。

**面试表达：** 短期记忆服务于当前任务，重点是准确、结构化和可恢复，不是把所有历史堆进 prompt。
### 3.2 长期记忆

**是什么：** 长期记忆是跨会话可复用的稳定信息，例如用户偏好、项目背景、历史修正、业务规则、常用工具选择。

**为什么需要：** 长期记忆可以提高个性化和连续性，但写入错误会长期污染模型行为。

**核心机制：** 长期记忆通常要经过候选、筛选、去重、冲突检测、置信度评估和生命周期管理。

**工程实现：** 工程上可用向量库保存语义记忆，用 PostgreSQL 保存结构化事实，用图数据库保存实体关系。读取时应基于当前任务相关性和权限过滤。

**常见坑：**

- 用户随口一说就写入长期记忆。
- 旧记忆和新事实冲突。
- 低置信记忆进入上下文。
- 无法删除或过期。

**优化方式：**

- 设置记忆候选区。
- 只写高置信、稳定、可复用信息。
- 写入前去重和冲突检测。
- 使用 TTL、版本和用户可管理机制。

**面试表达：** 长期记忆不是聊天记录仓库，而是经过治理的可复用知识。面试中要强调写入门槛、读取相关性和删除机制。
### 3.3 State 设计

**是什么：** State 是 Agent Workflow 中各节点共享和更新的任务状态。

**为什么需要：** 没有清晰 State，Agent 无法恢复、无法调试、无法判断下一步，也无法做节点级评测。

**核心机制：** State 应包含任务目标、当前步骤、计划、证据、工具结果、错误、风险、审批状态和最终输出等必要字段。

**工程实现：** LangGraph 中 State 可用 TypedDict 或 Pydantic 定义，节点只读取必要字段，只写自己负责的字段。复杂对象使用引用 ID。

**常见坑：**

- State 变成万能字典。
- 节点随意修改无关字段。
- 大文本堆积在 State。
- 没有 step_count、retry_count 等控制字段。

**优化方式：**

- 设计 State schema。
- 区分业务数据、过程数据和观测数据。
- 使用 reducer 处理列表追加或消息合并。
- 为关键字段记录版本和更新时间。

**面试表达：** State 设计决定 Agent 是否可控。一个好的 State schema 能让 Workflow 的路由、恢复、调试和评测都变得明确。
### 3.4 上下文压缩与选择

**是什么：** 上下文压缩与选择是从大量候选信息中挑选最相关、最可信、最必要的信息进入模型。

**为什么需要：** 上下文窗口不是越大越好。信息太多会增加成本、引入噪声，并导致模型忽略中间证据。

**核心机制：** 常见方式包括摘要、关键词提取、证据重排、去重、按任务阶段裁剪、按优先级放置首尾位置。

**工程实现：** 工程实现可设 token budget，将系统规则、用户目标、关键状态、证据和工具结果分层打包，每层有预算和优先级。

**常见坑：**

- 把所有 RAG 文档和工具结果全部拼接。
- 摘要丢失关键事实。
- 低相关记忆被注入。
- 证据顺序随意。

**优化方式：**

- 先按相关性和置信度筛选。
- 摘要保留原始引用。
- 关键信息放在模型更容易关注的位置。
- 对上下文构造做日志和评测。

**面试表达：** 上下文工程的重点不是填满窗口，而是在有限预算内给模型最有用、最可信、最可追溯的信息。
### 3.5 记忆读写策略

**是什么：** 记忆读写策略决定什么时候写入长期记忆，以及什么时候读取并注入上下文。

**为什么需要：** 没有策略的记忆系统会造成隐私风险、事实冲突和上下文污染。

**核心机制：** 写入策略关注稳定性、可复用性、置信度、权限和生命周期；读取策略关注相关性、精确性、时间有效性和当前任务必要性。

**工程实现：** 可以设计 memory_candidate、memory_validate、memory_write、memory_retrieve、memory_inject 等节点。用户敏感记忆需要可查看、可删除。

**常见坑：**

- 写入过多。
- 读取过宽。
- 没有用户控制权。
- 记忆和 RAG 证据冲突时不处理。

**优化方式：**

- 写入前评分。
- 读取时设置相似度阈值和 TopK。
- 冲突时优先最新、权威、显式来源。
- 记忆注入时标明来源和置信度。

**面试表达：** 记忆系统要保守：写入要慎重，读取要精确，注入要可解释。


## 4. 文字版架构图

### 4.1 上下文构造流程

```text
当前用户输入
  ↓
任务目标提取
  ↓
读取当前 State
  ↓
检索短期历史摘要
  ↓
按需读取长期记忆
  ↓
按需执行 RAG 检索
  ↓
合并工具结果
  ↓
权限与安全过滤
  ↓
相关性排序
  ↓
摘要 / 压缩 / 去重
  ↓
按优先级组装 Prompt
  ↓
模型调用
  ↓
输出解析
  ↓
更新 State / 记忆候选区
```

### 4.2 State / Memory / Evidence 分层

```text
State 层：
  当前任务状态、步骤、计划、错误、审批状态

Evidence 层：
  RAG chunk、工具返回、引用来源、原始文档 ID

Memory 层：
  用户偏好、长期事实、历史经验、项目背景

Context 层：
  本轮模型实际输入，由上面三层按需组合
```

## 5. 工程实战设计

### 5.1 LangGraph State 示例设计

```text
AgentState:
  task_id: str
  user_id: str
  user_goal: str
  current_step: str
  plan: list[dict]
  step_count: int
  retry_count: int
  retrieved_docs: list[doc_ref]
  tool_observations: list[observation_ref]
  memory_refs: list[memory_ref]
  risk_level: str
  approval_status: str
  errors: list[error]
  final_answer: str | None
```

设计原则：

```text
只放决策必需字段；
长文本用引用 ID；
列表字段明确追加规则；
关键控制字段必须存在；
节点只写自己负责字段。
```

### 5.2 长上下文治理策略

```text
任务拆分：
  大任务拆成子任务，每个子任务有独立上下文。

阶段摘要：
  每阶段结束生成结构化摘要，减少历史 token。

证据外部化：
  原始文档和工具结果放外部存储。

按需回取：
  需要时再检索原始证据。

预算控制：
  系统规则、用户目标、证据、记忆分别设置 token 上限。

一致性检查：
  最终答案检查是否与关键事实冲突。
```

## 7. 常见误区

### 7.1 把上下文窗口当作银弹

大上下文能减少截断，但不能自动解决噪声、排序、冲突和权限问题。上下文工程仍然需要选择和治理。

### 7.2 把记忆等同于历史聊天记录

长期记忆应该是经过抽取和验证的稳定事实，不是完整聊天日志。

### 7.3 忽视记忆隐私

用户偏好、身份、项目内容可能涉及隐私。记忆系统需要权限、可删除、可解释和最小化原则。

## 8. 面试题与答题闭环
### Q：Prompt Engineering 和 Context Engineering 有什么区别？

**考察点：** 概念边界、Agent 长任务。

**推荐回答：** Prompt Engineering 关注指令怎么写，包括角色、格式、约束和示例；Context Engineering 关注哪些信息进入模型，包括 RAG 证据、工具结果、历史摘要、状态和长期记忆如何选择、压缩、排序和淘汰。Agent 系统里 Context Engineering 更重要，因为它直接影响长任务稳定性。

### Q：LangGraph State 应该怎么设计？

**考察点：** 状态建模、可恢复、节点职责。

**推荐回答：** State 应只保存当前任务决策必需的信息，比如 task_id、目标、计划、步骤、工具结果引用、检索证据引用、错误、重试次数、风险和审批状态。长文本应外部化，State 保存引用 ID；节点只读必要字段，只写负责字段，避免万能字典和状态膨胀。

### Q：长期记忆如何避免污染？

**考察点：** 记忆治理、写入读取策略。

**推荐回答：** 不能把所有对话都写入长期记忆。要设候选区，按稳定性、置信度、可复用性和权限评估后再写入；写前去重和冲突检测，写后支持 TTL、版本和删除。读取时只取当前任务强相关且高置信的记忆。
